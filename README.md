# History Simulator

Интерактивный учебный проект по альтернативной истории России XIX–XX веков.

Пользователь выбирает исторический сценарий, проходит по развилкам решений и получает последствия каждого выбора в формате текстового квеста.

Проект состоит из трёх частей:

- `backend/` — API и игровая логика
- `frontend/` — интерфейс пользователя
- `ml/` — генерация текстовых ответов для узлов сценария

---

## Архитектура проекта

### Общая схема

```text
Frontend -> Backend -> ML
```

- **Frontend** показывает список сценариев, текущий узел, варианты выбора, текущий путь, карту дерева и связи между узлами.
- **Backend** хранит дерево сценариев, состояние сессии, текущий путь, посещённые узлы и правила переходов.
- **ML** возвращает текстовые ответы для текущего узла сценария.

## Как работает логика игры

Каждый сценарий описан как дерево узлов.

### Узел содержит

- `id`
- `title`
- `scene_text`
- список `choices`

### Каждый выбор содержит

- `id`
- `text`
- `next_node_id`

### Backend хранит для каждой сессии

- `current_node_id` — текущий узел
- `path_node_ids` — реальный путь, по которому игрок пришёл в текущую точку
- `visited_node_ids` — все узлы, которые уже были открыты в этой сессии

Это позволяет:

- идти вперёд по сценарию
- возвращаться назад по реальному пути
- переключаться в любой уже посещённый узел
- видеть карту дерева и текущую активную ветку

## Запуск backend

Из корня проекта:

```bash
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install -r backend/requirements.txt
python run_backend.py
```

Backend будет доступен на:

```text
http://127.0.0.1:8000
```

Swagger-документация:

```text
http://127.0.0.1:8000/docs
```

### 1. Запусти backend

```bash
source .venv/bin/activate

```

```bash
cd frontend
npm run dev
```

После этого:

- frontend открывается в браузере
- frontend отправляет запросы в backend
- backend использует `ml/generator.py` для текстовых ответов

## Основные API-ручки backend

### Получить список сценариев

```http
GET /api/scenarios
```

### Запустить сценарий

```http
POST /api/scenarios/start
Content-Type: application/json
```

Пример тела:

```json
{
  "scenario_id": "decembrist_uprising"
}
```

### Выполнить выбор

```http
POST /api/turn/choose
Content-Type: application/json
```

Пример тела:

```json
{
  "session_id": "SESSION_ID",
  "choice_id": "c1"
}
```

### Перейти в уже посещённый узел

```http
GET /api/node?session_id=SESSION_ID&node_id=NODE_ID
```

### Вернуться на шаг назад

```http
POST /api/turn/back?session_id=SESSION_ID
```

## Что получает frontend от backend

Frontend использует ответы backend, в которых есть:

- `current_node`
- `path_node_ids`
- `visited_node_ids`
- `tree_nodes`
- `scenario_title`
- `session_id`

Это позволяет фронту:

- отрисовать текущий экран сценария
- показать варианты выбора
- показать активный путь
- показать карту дерева и связи между узлами
- подсветить текущий узел, посещённые узлы и активную ветку

## Распределение работы по команде

### Backend

Backend-разработчик отвечает за:

- API
- хранение сессий
- структуру сценариев
- переходы между узлами
- текущий путь и посещённые узлы
- интеграцию с ML

### Frontend

Frontend-разработчик отвечает за:

- экран выбора сценария
- отображение текущего узла
- отображение вариантов выбора
- отображение пути
- отображение карты дерева и связей
- запросы к backend API

### ML

ML-разработчик отвечает за:

- генерацию текста ответа для узла
- улучшение исторической правдоподобности
- переход от заглушек к шаблонам, правилам или модели
- возможную генерацию новых веток и более свободного ввода

## Как дальше улучшать проект

### Backend идеи

- вынести сценарии из Python-файлов в JSON/YAML
- добавить сохранение сессий
- добавить свободный текст пользователя наряду с кнопками

- сделать ответы более исторически аргументированными
- учитывать историю предыдущих ходов
- генерировать разные типы последствий: политические, социальные, военные
- использовать шаблоны + LLM или rules + retrieval
- сделать ответы длиннее и контекстнее для ключевых узлов
